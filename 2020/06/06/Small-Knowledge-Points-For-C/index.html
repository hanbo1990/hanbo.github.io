Small Knowledge Points For C/C++

Posted on 2020-06-06 Edited on 2020-09-08 In C and Cpp

This page contains small knowledge points for C/C++ and will be updated randomly..

Keywords

const

  1. const var, var cannot be changed.
  2. pointer related
    • const pointer (const int *value) is pointing to const value.
    • pointer const (int *const value) means pointer itself is constant.
    • Part Before * is what the pointer is pointing to

inline

  1. Used before function definition, not declaration.
  2. Each inline call will introduce a copy of code, but the benefit is reduce the overhead of function call.
  3. Used for small and simple function which are called a lot to improve speed, and also allow type check.
  4. Should not contain complex condition code like while, switch. The inline function should not call itself.

volatile and barrier

It tells compiler don’t optimize the variable operation.

Compiler will do optimization for us.

Also, the optimization will make the code get the value from cache instead of the actual physical address. For example, when doing ADC, we need to add volatile to the value register to make sure the we are always using the latest value.

Adding volatile solve the problem that compiler will change the assembly code to operate a certain variable. But it cannot solve the problem that CPU will do this during runtime.

The famous problem is Singleton.

It seems correct. There is always a valid pInst in the return value. Double if can make sure the lock/unlock is as short as possible. But this piece of code may not work because of the CPU Out-of-order execution. In c++, new is done in 2 steps:

  1. Assign space
  2. Call constructor

So pInst = New T include 3 steps:

  1. Assign space
  2. Call constructor on the assigned space
  3. give the pointer value to pInst

It can happen that pInst is not NULL (3 happens first) before construction is not done (2 then happens). If at this moment, another thread is calling the GetInstance, pInst is not NULL. Then the pInst is return to the second call, oops, if another thread is accessing the data, crash may happen. The solution is barrier. The barrier will stop CPU from changing command after barrier.

Macros

debug

## is added in case the argument is empty.

do{}while(0) is added here, you can try to find why :).

# and ##

After preprocessor, the result is ("front" and "back" is aaa frontbackaa)

# will replace the symbol name of the input.

## provides a way to concatenate actual arguments during macro expansion. If a parameter in the replacement text is adjacent to a ##, the parameter is replaced by the actual argument, the ## and surrounding white space are removed, and the result is re-scanned.

In stm32 BSP, it’s used quite often.

#error

A typical use of the #error directive is to prevent compilation if a known condition exists that would cause the program to not function properly if the compilation completed. (For example, the C source was designed for a specific platform and was being compiled on an incompatible platform.)

#pragma

#pragma pack

#pragma pack instructs the compiler to pack structure members with particular alignment.

we can use pack to:

  1. Reserve some space for later usage because of the pad.
  2. Save space based on the CPU word size. If we pack the structure to 1 byte alignment, no pad (space waste) will exist. But this will make accessing this structure slower.

When creating a structure, we also need to think about arranging fields to save memory space. For example we are on a 32-bit system where the default byte alignment is 4.

#pragma message

Use #pragma message to specify a user-defined message within your program code. The message will be printed during compile time. For example, we can use this for todos.

#pragma warning

#pragma GCC warning “message” causes the preprocessor to issue a warning diagnostic with the text ‘ message ‘.

Functions

assert()

If the argument expression of this macro with functional form compares equal to zero (i.e., the expression is false), a message is written to the standard error device and abort is called, terminating the program execution.

  1. assert() is used in debug mode. Can be disabled in production code by #define NDEBUG before the first include of <assert.h>.
  2. It’s normally used to check if the input of functions are valid. It cannot be used to replace if checking. If is used to handled some conditions the input becomes invalid (maybe because of a bug).

_Static_assert (from c11)

The constant expression is evaluated at compile time and compared to zero. If it compares equal to zero, a compile-time error occurs and the compiler must display message (if provided) as part of the error message.

Other

Do not cast pointers into more strictly aligned pointer types

Example below may crash on some platform. Assume we are on a big endian system with word size 4 bytes, if the data is located on the second byte, the CPU need to have 2 bus read operation to write to the data and the 3 bytes after it. In this case some system may fail.

Some hardware architectures have relaxed requirements with regard to pointer alignment. Using a pointer that is not properly aligned is correctly handled by the architecture, although there might be a performance penalty. On such an architecture, improper pointer alignment is permitted but remains an efficiency problem.

Variable Default value

Normally global variables are initialized to 0. Local variable are quite random. The good practice is to initialize local variable if the value will be used. Because on some embedded systems, the software soft reset will not reset the memory, it’s also good to initialize the global variable in the module’s init function or reset function.

Reentrant Function and Thread Safety

Entering the same function twice only happens when:

  1. Multiple threads accessing the same function
  2. The function call itself.

To allow this, the reentrant function must:

  1. cannot use any local static variable or global non-const variable.
  2. does not return any local static variable or pointer to global non-const variable.
  3. does not depends on any input arguments.
  4. does not rely on lock on certain resources.
  5. does not call non-Reentrant Function.

If a function is reentrant, it’s safe to be used in multi-thread environment.
